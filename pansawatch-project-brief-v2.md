# PansaWatch.org — 프로젝트 브리프 v2

## 1. 프로젝트 개요

**PansaWatch.org**는 대한민국 법관(판사)의 공개 판결·뉴스·경력 정보를 수집·정리하고, 시민이 개별 판결에 대해 의견을 표현할 수 있는 **시빅테크(Civic Tech) 플랫폼**이다.

시빅테크란 기술을 활용해 시민의 공적 참여를 돕거나 공공 영역의 투명성을 높이는 활동을 말한다. PansaWatch는 흩어져 있는 사법부 공개 정보를 한 곳에 모아 시민이 쉽게 접근할 수 있게 만드는 정보 집계 플랫폼이다.

### 레퍼런스

일본의 裁判官マップ(재판관맵, https://saibankan-map.jp)을 참고한다. 이 서비스는 변호사 다나카 카즈야가 Claude Code를 활용해 약 1개월 만에 단독 개발했으며, 전국 약 2,500명의 재판관 정보를 지도 기반으로 제공하고, 판례 AI 요약과 구두후기(리뷰·별점) 기능을 갖추고 있다. 2026년 3월 공개 후 하루 2~3만 명이 방문할 정도로 화제가 되었다.

### PansaWatch의 차별점

- 일본 재판관맵의 **판사 개인 별점/리뷰 방식을 채택하지 않는다**. 대신 **판결 단위의 시민 투표**를 핵심 기능으로 설계한다. 판사 개인이 아닌 공문서(판결)에 대한 의견 표현이므로 법적으로 훨씬 안전하면서, 판결별 투표가 축적되면 결과적으로 해당 판사에 대한 시민 평가가 자연스럽게 집계된다.
- 판사별 관련 **뉴스 기사**를 자동 수집하여 헤드라인 + 링크 + AI 요약으로 제공한다.
- 판사별 **객관적 통계**(항소심 파기율, 사건유형별 비율 등)를 산출하여 표시한다.
- 운영자는 편집 판단을 하지 않으며, 기사 수집은 키워드 기반 자동화로 중립성을 보장한다.

## 2. 법적 분석

### 2-1. 핵심 설계 원칙: "판사"가 아니라 "판결"을 평가한다

"김OO 판사 별점 2점"은 사람에 대한 평가이므로, 해당 판사가 명예훼손을 주장하거나 임시조치를 요청할 명분이 생긴다. 반면 "2024가합12345 사건 판결에 동의하십니까?"는 국가가 공개한 공문서에 대한 의견 표현이다. 판결은 공적 행위이며, 이에 대한 시민의 찬반은 표현의 자유의 핵심 영역에 해당한다.

판결 단위 투표가 축적되면, 예를 들어 "담당 판결 20건 중 시민 동의율 35%"처럼 판사에 대한 평가가 자연스럽게 드러난다. 별점을 직접 매기지 않아도 동일한 효과가 나면서, 법적으로는 구조적으로 안전하다.

### 2-2. 플랫폼 각 구성요소의 법적 안전성

**판사 프로필 (이름, 소속, 직위, 경력)**
대법원이 매년 공개하는 법관 인사발령 데이터이며, 판결문에서도 법관 실명이 유지된다(비실명 처리 대상에서 제외). 정부가 공표한 공적 정보이므로 명예훼손 대상이 되지 않는다.

**뉴스 기사 링크 + AI 요약**
이미 언론사가 보도한 내용의 링크와 요약이다. 기사 원문을 복제하지 않고 링크 + AI 요약만 제공하여 저작권 리스크를 회피한다. 기사 수집은 키워드 기반 자동화이므로 운영자의 편집 의도가 개입되지 않는다.

**판결문 정보 + AI 요약**
판결문은 국가가 공개한 공문서이다. 공문서의 요약을 명예훼손으로 주장할 근거가 없다.

**판결에 대한 시민 투표 결과**
"시민 68%가 이 판결에 동의하지 않음"은 여론조사 결과와 같은 성격의 의견 집계이다. 사실의 적시도 아니고 의견의 집계이므로 명예훼손 대상이 되기 어렵다.

**객관적 통계 (항소심 파기율, 사건유형별 비율 등)**
판결문 데이터에서 추출한 팩트(fact)이다. 통계 자체에 대해 명예훼손을 주장하기는 극히 어렵다.

### 2-3. 임시조치 대응

정보통신망법 제44조의2에 따른 임시조치는 **"해당 정보" 단위**로 적용된다. 판사 프로필 페이지 전체가 아니라, 특정 게시물(정보)을 지목해야 한다. PansaWatch의 각 구성요소는 모두 공적 데이터(인사발령, 판결문)이거나 이미 공개된 보도(뉴스 링크)의 집계이므로, 임시조치를 요청할 수 있는 타겟이 구조적으로 거의 없다.

참고: 국가인권위원회는 2022년 12월 "임시조치 대상 정보가 공적 인물에 관한 사안이나 공공의 관심 사안에 해당하는 경우 임시조치 대상에서 제외하도록" 정보통신망법 개정을 권고한 바 있다(아직 법 개정은 미완).

### 2-4. 참고 판례

**나무위키 학교법인 소송 (2026년)**
어떤 학교 법인이 나무위키에 스쿨미투·사학비리·부당해고 의혹 등이 기재된 문서에 대해 500만 원 손해배상과 삭제를 요청하는 소송을 제기했으나, 법원은 "게시물의 불법성이 명백하다고 보기에 부족하고, 가치중립적 성격을 띠고 있어 구체적인 피해를 입었다고 보기 어렵다"며 청구를 모두 기각했다.

이 판례는 PansaWatch에 직접적 시사점을 준다. 공개된 사실을 가치중립적으로 정리해놓은 것 자체는 명예훼손이 아니라는 것이 법원의 판단이다.

### 2-5. 저작권

- 기사 원문을 저장하지 않는다. 제목 + URL + AI 요약만 보관한다.
- AI 요약문 하단에 "AI가 생성한 요약이며, 정확한 내용은 원문을 확인하세요" 면책 문구를 포함한다.

## 3. 현재 진행 단계

> **현재 위치는 "Phase 1 목업 프로토타입 완료"가 아니라, 데이터/인프라는 Phase 3 수준까지 진척되었으나 핵심 투표 기능은 미완인 혼재 상태이다.**

- **판사/법원 데이터**: 이미 실데이터로 구축됨 (판사 2,987명·법원 83곳). 브리프 §7의 "30~50명 목업"과 다르며, 실데이터 단계로 이행한 상태이다.
- **뉴스 수집 파이프라인**: 일부 가동 중(자체 GPU 기반, 수동 실행). 자동화·DB 연동은 미완이다.
- **⭐ 핵심 기능 "판결 단위 시민 투표": 현재 비작동.** cases/caseVotes/articleVotes 데이터 0건, 투표 위젯 미표시, 영속화(인증·API·DB) 미구현 상태이다.

## 5. 데이터 모델 (핵심 테이블)

### judges (판사)

```
id: UUID (PK)
name: string — 판사 이름
court: string — 소속 법원 (예: "서울중앙지방법원")
court_region: string — 지역 (예: "서울")
position: string — 직위 (예: "부장판사", "판사")
appointment_year: number — 임관 연도
career_summary: text — 약력 요약
photo_url: string | null — 사진 URL (공개된 경우)
created_at: timestamp
updated_at: timestamp
```

### courts (법원)

```
id: UUID (PK)
name: string — 법원 이름
type: enum — "supreme" | "high" | "district" | "family" | "administrative"
region: string — 지역
address: string — 주소
latitude: number — 위도
longitude: number — 경도
judge_count: number — 소속 판사 수
```

### articles (뉴스 기사)

```
id: UUID (PK)
title: string — 기사 헤드라인
url: string — 원문 링크
source: string — 언론사 (예: "한겨레", "조선일보")
published_at: timestamp — 기사 발행일
ai_summary: text — AI 생성 요약 (2~3문장)
collected_at: timestamp — 수집 시각
```

### judge_articles (판사-기사 매핑, 다대다)

```
id: UUID (PK)
judge_id: UUID (FK → judges)
article_id: UUID (FK → articles)
relevance_score: number — 연관도 점수 (0~1)
```

### cases (판례)

```
id: UUID (PK)
case_number: string — 사건번호 (예: "2024가합12345")
court: string — 법원명
judge_id: UUID (FK → judges)
case_type: string — 사건 유형 (민사/형사/행정/가사)
decision_date: date — 선고일
decision_result: string — 판결 결과 (예: "원고 승소", "피고 승소", "일부 승소")
ai_summary: text — AI 요약
source_url: string — 판례 원문 링크
is_appealed: boolean — 항소 여부
appeal_result: string | null — 항소심 결과 (원심유지/파기환송/파기자판 등)
```

### case_votes (판결에 대한 시민 투표) ⭐ 핵심

```
id: UUID (PK)
user_id: UUID (FK → users)
case_id: UUID (FK → cases)
vote_category: enum — "decision_agreement" | "sentencing_fairness"
vote_value: enum — "agree" | "disagree" (decision_agreement용)
                   "appropriate" | "too_light" | "too_heavy" (sentencing_fairness용)
created_at: timestamp
UNIQUE(user_id, case_id, vote_category)
```

- decision_agreement: "이 판결에 동의하십니까?" (동의/비동의)
- sentencing_fairness: "양형이 적절하다고 생각하십니까?" (적절/과소/과중) — 형사 사건에만 표시

### article_votes (기사에 대한 투표)

```
id: UUID (PK)
user_id: UUID (FK → users)
article_id: UUID (FK → articles)
vote_type: enum — "useful" | "not_useful"
created_at: timestamp
UNIQUE(user_id, article_id)
```

### users (이용자)

```
id: UUID (PK)
auth_provider: string — "google" | "kakao"
created_at: timestamp
```

## 6. 페이지 구조 및 UI 설계

### 6-1. 메인 페이지 (`/`)

- **상단**: PansaWatch.org 로고 + 검색바 (판사 이름, 법원명, 사건번호로 검색)
- **중앙**: 대한민국 지도
  - 지도 위에 각 법원 위치를 마커로 표시
  - 마커 크기 또는 색상으로 소속 판사 수 표현
  - 마커 클릭 → 해당 법원의 판사 목록 팝업/페이지 이동
- **하단**: 최근 수집된 뉴스 기사 피드 (최신 10~20건)

### 6-2. 법원 상세 페이지 (`/courts/[id]`)

- 법원 기본 정보 (이름, 유형, 주소, 지도 위치)
- 소속 판사 목록 (카드 형태)
  - 각 카드: 이름, 직위, 담당 판결 수, 시민 동의율

### 6-3. 판사 상세 페이지 (`/judges/[id]`) ⭐ 핵심 페이지

- **프로필 섹션**
  - 이름, 소속 법원, 직위, 임관 연도, 경력 요약
  - **요약 통계 카드**: "담당 판결 N건 · 시민 동의율 XX% · 관련 기사 N건"
  - 이 통계는 운영자의 의견이 아닌 이용자 투표와 데이터의 기계적 집계임

- **판례 탭** (기본 탭)
  - 담당 판례 리스트 (최신순)
  - 각 판례 카드:
    - 사건번호 + 사건유형 배지 + 선고일
    - AI 요약 (3~5문장)
    - **시민 투표 영역**: "이 판결에 동의하십니까?" (동의/비동의 버튼 + 현재 비율 표시)
    - 형사 사건의 경우 추가: "양형이 적절하다고 생각하십니까?" (적절/과소/과중)
    - 원문 링크 버튼
  - 면책 문구: "AI가 생성한 요약이며, 정확한 내용은 원문을 확인하세요"

- **뉴스 탭**
  - 관련 기사 리스트 (최신순)
  - 각 기사: 헤드라인(원문 링크) + 언론사 + 날짜 + AI 요약
  - 기사별 투표: "이 기사가 유용했나요?" (유용함/유용하지 않음)

- **통계 탭**
  - 판결별 시민 동의율 차트 (시계열)
  - 사건유형별 비율 (파이차트)
  - 항소심 파기율 (전국 평균 대비)
  - 관련 기사 수 추이 (월별)

### 6-4. 뉴스 피드 페이지 (`/news`)

- 전체 수집 기사 목록 (최신순, 페이지네이션)
- 필터: 언론사, 기간, 법원 지역

### 6-5. 소개 페이지 (`/about`)

- 프로젝트 소개 및 취지 (사법 투명성, 시민 참여)
- 데이터 수집 방법 투명 공개 (수집 키워드, 크롤링 주기, 대상 사이트)
- 법적 근거 및 면책 조항
- 비영리 선언 (광고 없음, 수익 모델 없음)
- 문의처

## 7. Mock 데이터 생성 가이드

현재 단계에서는 실제 크롤링 없이 가상 데이터를 사용한다. 아래 규모로 생성한다:

### 판사 (30~50명)

- 실제 법원 이름과 구조를 반영 (서울중앙지방법원, 서울고등법원, 부산지방법원 등)
- 이름은 가상 (예: 김민수, 이정아, 박현우 등)
- 직위: 대법관, 고등법원 부장판사, 지방법원 판사 등 현실적 분포

### 법원 (15~20곳)

실제 대한민국 법원 목록에 기반:
- 대법원 (서초구)
- 서울고등법원

## 8. 디자인 방향

### 톤 & 무드

- **에디토리얼 + 시빅**: 신뢰감 있는 미디어/보고서 느낌
- 색상: 네이비 + 화이트 기반, 강조색은 앰버 또는 코랄 (경고/중요 정보용)
- 과도한 장식 배제, 정보 밀도 높게 (데이터 중심 UI)
- 모바일 반응형 필수

### 디자인 참고

- 일본 재판관맵 (https://saibankan-map.jp/judges/map) — 지도 기반 탐색 UI
- OpenSecrets.org — 정치 데이터 시빅테크의 정보 구조
- 열려라 국회 (https://watch.peoplepower21.org/) — 한국 시빅테크 레퍼런스

### 핵심 UX 원칙

1. **검색이 최우선** — 방문자는 특정 판사 이름이나 사건번호를 검색해서 들어오는 경우가 가장 많다
2. **지도는 탐색용** — 모르는 판사를 발견하는 인터페이스
3. **기사 → 원문 링크 유도** — 사이트 자체에서 기사를 읽게 하지 않고, 원문으로 보낸다
4. **투표는 판결 단위** — 판사 개인이 아니라 "이 판결에 동의하는가" 단위로 투표하고, 결과는 판사 프로필에 자동 집계된다

## 9. 개발 순서 (우선순위)

> **현재 실제 위치**: 데이터/인프라(판사·법원 실데이터, 뉴스 수집 일부)는 Phase 3 수준까지 진척되었으나, Phase 1의 핵심인 판결 단위 시민 투표는 미완이라 단계가 혼재되어 있다.

### Phase 1: 프론트엔드 프로토타입

> **미완료.** 핵심 기능인 판결 단위 시민 투표 데이터(cases/caseVotes/articleVotes)와 영속화(인증·API·DB)가 빠져 있어 Phase 1은 아직 완료되지 않았다.

1. Next.js 프로젝트 초기화 (App Router, TypeScript, Tailwind)
2. Mock 데이터 JSON 파일 생성 (판사, 법원, 기사, 판례, 투표)
3. 메인 페이지 — 대한민국 지도 + 검색바 + 최신 뉴스 피드
4. 판사 상세 페이지 — 프로필 + 요약통계 + 판례탭(투표 포함) + 뉴스탭 + 통계탭
5. 법원 상세 페이지 — 법원 정보 + 소속 판사 카드 (동의율 포함)
6. 뉴스 피드 페이지
7. About 페이지
8. 모바일 반응형 적용

### Phase 2: 백엔드 + DB

1. Supabase 프로젝트 생성 및 스키마 구축
2. Mock 데이터를 DB로 마이그레이션
3. API 엔드포인트 구현 (판사 조회, 판례 조회, 투표 CRUD)
4. Supabase Auth 연동 (투표 기능용)
5. 투표 집계 로직 구현

### Phase 3: 데이터 수집 파이프라인

1. 뉴스 크롤러 (네이버 뉴스 API + 구글 뉴스 RSS)
2. Claude Haiku 기사 요약 파이프라인
3. 판사-기사 매칭 로직 (NER + 규칙 기반)
4. AWS Lambda + EventBridge 스케줄러

### Phase 4: 판례 데이터 연동

1. 대법원 종합법률정보 판례 수집
2. Claude Sonnet 판례 요약
3. 판사-판례 연결
4. 항소심 파기율 등 통계 산출

### Phase 5: 고도화

1. Elasticsearch 검색 도입
2. 통계 차트 고도화 (시계열, 비교 분석)
3. SEO 최적화
4. 성능 최적화 (캐싱, ISR)
5. 도메인 연결 (pansawatch.org)

## 12. 참고 사항

- **개인정보**: 판사의 개인 연락처, 주소 등 사적 정보는 수집하지 않는다. 공적 직무 정보만 다룬다.
- **면책 문구**: 모든 AI 요약에 "이 요약은 AI가 생성한 것이며, 정확한 내용은 원문을 참조하세요"를 표시한다.
- **데이터 수집 투명성**: /about 페이지에서 수집 키워드, 크롤링 주기, 수집 대상 사이트를 모두 공개한다.
- **비영리 포지셔닝**: .org 도메인 사용, 광고 없음, 수익 모델 없음을 명시한다.
- **투표 무결성**: 투표는 로그인 필수 (Google/Kakao OAuth), 판결당 1인 1투표, 투표 변경 가능.
