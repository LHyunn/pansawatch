# PansaWatch.org — 프로젝트 브리프

## 1. 프로젝트 개요

**PansaWatch.org**는 대한민국 법관(판사)에 대한 공개 뉴스와 판례 정보를 수집·정리하여 시민이 쉽게 열람할 수 있도록 하는 **시빅테크(Civic Tech) 플랫폼**이다.

### 레퍼런스

일본의 裁判官マップ(재판관맵, https://saibankan-map.jp)을 참고한다. 이 서비스는 변호사 다나카 카즈야가 Claude Code를 활용해 약 1개월 만에 단독 개발했으며, 전국 약 2,500명의 재판관 정보를 지도 기반으로 제공하고, 판례 AI 요약과 구두후기(리뷰) 기능을 갖추고 있다.

### PansaWatch의 차별점

- 리뷰/평점 기능은 제외한다 (법적 리스크 회피).
- 대신 **판사별 관련 뉴스 기사**를 자동 수집하여 헤드라인 + 링크 + AI 요약으로 제공한다.
- 방문자가 기사를 직접 읽고 **투표**(유용함/유용하지 않음 등)할 수 있다.
- 운영자는 편집 판단을 하지 않으며, 기사 수집은 키워드 기반 자동화로 중립성을 보장한다.

### 법적 안전성 요약

- 판사는 공적 인물이며, 판결문에서 법관 실명이 공개된다.
- 운영자가 의견을 표현하지 않고 공개 기사를 기계적으로 수집·정리하는 구조이므로 명예훼손 리스크가 극히 낮다.
- 기사 원문을 복제하지 않고, 링크 + AI 요약만 제공하여 저작권 리스크를 회피한다.
- AI 요약문 하단에 "AI가 생성한 요약이며, 정확한 내용은 원문을 확인하세요" 면책 문구를 포함한다.
- 투표는 판사 개인이 아닌 판결·사건 단위로 설계한다 ("이 판결이 적절하다고 생각하나요?").

## 3. 기술 스택

| 계층 | 기술 | 비고 |
|------|------|------|
| 프론트엔드 | Next.js 14+ (App Router) + React + TypeScript | SSR로 SEO 확보 |
| 스타일링 | Tailwind CSS | 유틸리티 기반 |
| 지도 | react-simple-maps 또는 D3.js | 대한민국 지도 렌더링 |
| 백엔드 | Next.js API Routes | 프론트와 동일 프로젝트 |
| 데이터베이스 | PostgreSQL (Supabase) | 추후 연동 |
| 인증 | Supabase Auth (Google OAuth) | 투표 시 사용 |
| 검색 | Elasticsearch | 추후 연동 |
| 캐시 | Redis | 추후 연동 |
| AI 요약 (기사) | Claude Haiku | 빠르고 저렴 |
| AI 요약 (판례) | Claude Sonnet | 법률 전문성 필요 |
| 배포 | Vercel | 무료 플랜으로 시작 |
| 크롤러 | AWS Lambda + EventBridge | 하루 1~2회 스케줄 |
| 파일 저장 | AWS S3 | 정적 파일 |

## 4. 데이터 모델 (핵심 테이블)

### judges (판사)

```
id: UUID (PK)
name: string — 판사 이름
court: string — 소속 법원 (예: "서울중앙지방법원")
court_region: string — 지역 (예: "서울")
position: string — 직위 (예: "부장판사", "판사")
appointment_year: number — 임관 연도
career_summary: text — 약력 요약
photo_url: string | null — 사진 URL (공개된 경우)
created_at: timestamp
updated_at: timestamp
```

### courts (법원)

```
id: UUID (PK)
name: string — 법원 이름
type: enum — "supreme" | "high" | "district" | "family" | "administrative"
region: string — 지역
address: string — 주소
latitude: number — 위도
longitude: number — 경도
judge_count: number — 소속 판사 수
```

### articles (뉴스 기사)

```
id: UUID (PK)
title: string — 기사 헤드라인
url: string — 원문 링크
source: string — 언론사 (예: "한겨레", "조선일보")
published_at: timestamp — 기사 발행일
ai_summary: text — AI 생성 요약 (2~3문장)
collected_at: timestamp — 수집 시각
```

### judge_articles (판사-기사 매핑, 다대다)

```
id: UUID (PK)
judge_id: UUID (FK → judges)
article_id: UUID (FK → articles)
relevance_score: number — 연관도 점수 (0~1)
```

### cases (판례)

```
id: UUID (PK)
case_number: string — 사건번호 (예: "2024나12345")
court: string — 법원명
judge_id: UUID (FK → judges)
case_type: string — 사건 유형 (민사/형사/행정/가사)
decision_date: date — 선고일
ai_summary: text — AI 요약
source_url: string — 판례 원문 링크
```

### votes (이용자 투표)

```
id: UUID (PK)
user_id: UUID (FK → users)
article_id: UUID (FK → articles)
vote_type: enum — "agree" | "disagree"
created_at: timestamp
UNIQUE(user_id, article_id)
```

## 5. 페이지 구조 및 UI 설계

### 5-1. 메인 페이지 (`/`)

- **상단**: PansaWatch.org 로고 + 검색바 (판사 이름, 법원명으로 검색)
- **중앙**: 대한민국 지도
  - 지도 위에 각 법원 위치를 마커로 표시
  - 마커 크기 또는 색상으로 소속 판사 수 표현
  - 마커 클릭 → 해당 법원의 판사 목록 팝업/페이지 이동
- **하단**: 최근 수집된 뉴스 기사 피드 (최신 10~20건)

### 5-2. 법원 상세 페이지 (`/courts/[id]`)

- 법원 기본 정보 (이름, 유형, 주소, 지도 위치)
- 소속 판사 목록 (카드 형태)
  - 각 카드: 이름, 직위, 관련 기사 수

### 5-3. 판사 상세 페이지 (`/judges/[id]`) ⭐ 핵심 페이지

- **프로필 섹션**: 이름, 소속 법원, 직위, 임관 연도, 경력 요약
- **관련 뉴스 탭**:
  - 기사 리스트 (최신순)
  - 각 기사: 헤드라인(링크) + 언론사 + 날짜 + AI 요약 + 투표 버튼
  - "AI가 생성한 요약이며, 정확한 내용은 원문을 확인하세요" 면책 문구
- **판례 탭**:
  - 담당 판례 리스트
  - 각 판례: 사건번호 + 사건유형 + 선고일 + AI 요약 + 원문 링크
- **통계 탭** (추후):
  - 관련 기사 수 추이 (월별 차트)
  - 사건 유형별 비율

### 5-4. 뉴스 피드 페이지 (`/news`)

- 전체 수집 기사 목록 (최신순, 페이지네이션)
- 필터: 언론사, 기간, 법원 지역

### 5-5. 소개 페이지 (`/about`)

- 프로젝트 소개 및 취지
- 데이터 수집 방법 투명 공개 (키워드 기반 자동 수집임을 명시)
- 면책 조항
- 문의처

## 6. Mock 데이터 생성 가이드

현재 단계에서는 실제 크롤링 없이 가상 데이터를 사용한다. 아래 규모로 생성한다:

### 판사 (30~50명)

- 실제 법원 이름과 구조를 반영 (서울중앙지방법원, 서울고등법원, 부산지방법원 등)
- 이름은 가상 (예: 김민수, 이정아, 박현우 등)
- 직위: 대법관, 고등법원 부장판사, 지방법원 판사 등 현실적 분포

### 법원 (15~20곳)

실제 대한민국 법원 목록에 기반:
- 대법원 (서초구)
- 서울고등법원
- 서울중앙지방법원, 서울동부지방법원, 서울남부지방법원, 서울북부지방법원, 서울서부지방법원
- 수원지방법원, 인천지방법원
- 대전지방법원, 대구지방법원, 부산지방법원, 광주지방법원
- 울산지방법원, 제주지방법원
- 각 법원의 실제 위도/경도 좌표 사용

### 뉴스 기사 (100~150건)

- 실제 존재할 법한 헤드라인 생성 (예: "서울중앙지법 김OO 판사, 대형 횡령 사건 1심 선고")
- 실제 언론사 이름 사용 (한겨레, 조선일보, 연합뉴스, 법률신문, SBS, KBS 등)
- URL은 `#` 또는 `https://example.com/...` 플레이스홀더
- AI 요약문은 2~3문장으로 현실적 내용 생성
- 날짜는 최근 6개월 범위 내

### 판례 (50~80건)

- 현실적 사건번호 형식 (2024가합12345, 2025노1234 등)
- 사건유형: 민사, 형사, 행정, 가사 등
- AI 요약은 3~5문장

## 7. 디자인 방향

### 톤 & 무드

- **에디토리얼 + 시빅**: 신뢰감 있는 미디어/보고서 느낌
- 색상: 네이비 + 화이트 기반, 강조색은 앰버 또는 코랄 (경고/중요 정보용)
- 과도한 장식 배제, 정보 밀도 높게 (데이터 중심 UI)
- 모바일 반응형 필수

### 디자인 참고

- 일본 재판관맵 (https://saibankan-map.jp/judges/map) — 지도 기반 탐색 UI
- OpenSecrets.org — 정치 데이터 시빅테크의 정보 구조
- 열려라 국회 (https://watch.peoplepower21.org/) — 한국 시빅테크 레퍼런스

### 핵심 UX 원칙

1. **검색이 최우선** — 방문자는 특정 판사 이름을 검색해서 들어오는 경우가 가장 많다
2. **지도는 탐색용** — 모르는 판사를 발견하는 인터페이스
3. **기사 → 원문 링크 유도** — 사이트 자체에서 기사를 읽게 하지 않고, 원문으로 보낸다
4. **투표는 기사 단위** — 판사 개인이 아니라 "이 판결/기사가 유용한가" 단위

## 8. 개발 순서 (우선순위)

### Phase 1: 프론트엔드 프로토타입 (현재 단계) ✅

1. Next.js 프로젝트 초기화 (App Router, TypeScript, Tailwind)
2. Mock 데이터 JSON 파일 생성
3. 메인 페이지 — 대한민국 지도 + 검색바 + 최신 뉴스 피드
4. 판사 상세 페이지 — 프로필 + 뉴스 탭 + 판례 탭
5. 법원 상세 페이지 — 법원 정보 + 소속 판사 카드
6. 뉴스 피드 페이지
7. About 페이지
8. 모바일 반응형 적용

### Phase 2: 백엔드 + DB

1. Supabase 프로젝트 생성 및 스키마 구축
2. Mock 데이터를 DB로 마이그레이션
3. API 엔드포인트 구현
4. Supabase Auth 연동 (투표 기능용)

### Phase 3: 데이터 수집 파이프라인

1. 뉴스 크롤러 (네이버 뉴스 API + 구글 뉴스 RSS)
2. Claude Haiku 기사 요약 파이프라인
3. 판사-기사 매칭 로직
4. AWS Lambda + EventBridge 스케줄러

### Phase 4: 판례 데이터 연동

1. 대법원 종합법률정보 판례 수집
2. Claude Sonnet 판례 요약
3. 판사-판례 연결

### Phase 5: 고도화

1. Elasticsearch 검색 도입
2. 기사 수 추이 차트 등 통계 기능
3. SEO 최적화
4. 성능 최적화 (캐싱, ISR)
5. 도메인 연결 (pansawatch.org)

## 10. 참고 사항

- **저작권**: 기사 원문을 저장하지 않는다. 제목 + URL + AI 요약만 보관한다.
- **개인정보**: 판사의 개인 연락처, 주소 등 사적 정보는 수집하지 않는다. 공적 직무 정보만 다룬다.
- **면책 문구**: 모든 AI 요약에 "이 요약은 AI가 생성한 것이며, 정확한 내용은 원문을 참조하세요"를 표시한다.
- **데이터 수집 투명성**: /about 페이지에서 수집 키워드, 크롤링 주기, 수집 대상 사이트를 모두 공개한다.
- **비영리 포지셔닝**: .org 도메인 사용, 광고 없음, 수익 모델 없음을 명시한다.
